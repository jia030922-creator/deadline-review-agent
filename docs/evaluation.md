# 确定性验收规则说明

本文说明 Deadline Review Agent 当前规则模式如何使用用户提交说明、证据链接记录和真实上传文件。规则模式追求可解释和可重复，不尝试替代人工完成复杂语义判断。

## 三种单项状态

### PASS

`PASS` 表示当前可读取证据足以直接验证标准。例如：

- 标准要求 PDF，实际上传了可被 pypdf 识别的 PDF；
- 标准要求不超过一页，PDF 页数成功读取且为一页；
- 标准要求合法 JSON，上传的 JSON 已通过语法解析；
- 标准要求出现一组明确关键词，成功提取的文件文本包含全部关键词；
- 标准本身要求填写或说明某项内容，提交说明中存在直接、肯定且相关的文本证据。

`PASS` 不表示 Agent 已验证链接内容。证据链接只被记录，当前不会联网打开。

### FAIL

`FAIL` 表示存在可确定的反向证据或明确冲突。例如：

- 标准要求一页，实际 PDF 是两页；
- 标准要求 PDF，但只上传了 TXT；
- 标准要求合法 JSON，但上传的 JSON 存在语法错误；
- 文件文本已成功提取，但明确缺少标准列出的必需关键词；
- 提交说明明确写明某项未完成或缺失。

单项 `FAIL` 通常聚合为最终 `NEEDS_REVISION`，引导提交者修正后再次交付。明显空提交可直接聚合为最终 `FAIL`。

### NEEDS_REVIEW

`NEEDS_REVIEW` 表示当前证据不足以可靠确认通过或失败。例如：

- 用户声称提交了 PDF，但没有上传实际文件；
- PDF 损坏、加密或页数读取失败；
- PDF 页数可读，但没有可提取文本，而标准要求验证正文内容；
- 只有证据链接，无法读取链接中的内容；
- 标准要求判断创新性、商业价值、专业程度、整体美观或优秀代码质量；
- 标准自然语言过于复杂，超出有限规则的可靠范围。

`NEEDS_REVIEW` 不是失败，而是明确要求人工补证或复核。

## 证据优先级

当前优先级为：

1. 成功或部分成功解析的真实文件内容；
2. 文件解析得到的结构化信息，例如格式、PDF 页数和 JSON 合法性；
3. 用户提交说明中的声明；
4. 证据链接记录。

`submission_text` 只是用户声明。对于文件可验证标准，它不能提前把结果短路为 `PASS`，也不能覆盖成功解析文件给出的 `FAIL`。没有文件时，自述“已经包含某关键词”“是一页 PDF”或“JSON 合法”通常只能得到 `NEEDS_REVIEW`。

真实文件与用户自述冲突时，以文件解析结果为准。例如用户说明“简历已经控制在一页”，而实际 PDF 为两页，单项必须判定为 `FAIL`，理由会明确指出冲突。

解析失败的文件不具有这种覆盖能力。Agent 不会把扩展名为 `.pdf` 但无法解析的文件假装成有效 PDF，也不会把未提取到的正文当作已经检查。

文件内容缺少关键词只有在相关文件为 `SUCCESS`、文本未截断、关键词规则可靠且确实未命中时才判定 `FAIL`。如果文件为 `PARTIAL`、`FAILED`、`UNSUPPORTED`，正文为空或文本已截断，未命中只能得到 `NEEDS_REVIEW`。如果 `PARTIAL` 文件的已提取部分确实找到了全部关键词，则可以依据该明确命中判定 `PASS`。

## 可以确定性验证的标准

规则模式适合验证边界清楚、可直接观测的要求：

- 是否上传可识别的 PDF；
- PDF 是否不超过指定的简单页数，例如一页或不超过两页；
- JSON 是否语法合法；
- 已提取文本是否包含明确列出的词语或短语；
- 提交说明型标准中是否存在直接完成陈述或明确未完成陈述；
- 是否按截止时间提交。

关键词验证要求标准使用相对明确的形式，例如“包含 Deadline Box”或“必须出现安装步骤”。规则会检查列出的词语，不会把整句标准与全文做简单完全匹配。

## 必须人工复核的标准

以下类型不能由当前规则可靠完成：

- 内容是否有创新性；
- 项目是否具有较强商业价值；
- 文案是否足够专业；
- 代码质量是否优秀；
- 视觉是否美观；
- 需要理解上下文、推理隐含含义或比较多个方案的标准；
- 需要读取图片、扫描件、Word、Excel、音视频或网页内容的标准。

## 已知边界

- 仅支持 PDF、TXT、Markdown 和 JSON。
- PDF 使用 pypdf 提取文本，不包含 OCR；扫描型 PDF 可能只能得到页数。
- 每个文件最大 5 MB，提取文本最多保留 20,000 字符。
- 多个文件的关键词会在已成功提取的文本集合中检查；规则不会理解复杂的跨文件关系。
- 简单页数规则支持阿拉伯数字和常用中文数字，但不覆盖任意自然语言表达。
- 文件名和 MIME 类型仅作为元数据，扩展名不等于内容已验证。
- 证据链接不会被联网打开，也不会抓取网页或调用 GitHub、云盘 API。
- 链接地址或链接文字中出现关键词，不代表链接目标内容已被验证。
- 日志不保存原始文件二进制，只保存元数据和不超过 1,000 字符的文件文本预览。
